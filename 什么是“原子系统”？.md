原子应用，原子内存，原子隐私...近些年，各大品牌厂商都在疯狂宣发自己的“原子项目”。不过，相较与20年前的一篇计算机博士论文，以及现在的各大GNU/Linux发行版，这些概念却显得尤为落后。今天，就让我们来聊聊“原子系统” (Atomic Operating System)。事先声明：我们所聊到的“原子”是指操作系统层面上的原子化，前面提到的那些“噱头”，我们这里并不会作太多详细介绍。那么，就让我们走进操作系统的发展浪潮，来看看这个“时代前沿”究竟是什么。

## 1.原子，从何而来？(Nix)
首先，问各位一个问题：在安装软件时，各位有没有遇到过“依赖缺失”，“依赖冲突”，“文件不存在”等各种稀奇古怪的提示？如果有，那么恭喜你：你可能遇到了传说中的“依赖地狱（Dependency Hell）”。

2003年，一位叫做Eelco Dolstra的年轻人在使用Linux时，也敏锐地察觉到了Linux在包管理方面的局限性。大多数操作系统在依赖冲突问题上的处理及其不完备（也可以说压根没有什么处理），在安装某些软件包时，部分软件可能会因为依赖库的版本需求不同而无法共存，这对于开发者与用户及其不便:
> Similarly, RPM does not allow two packages that contain files with equal path names to exist simultaneously in the system. 
> 
> ...
> 
> In general, this means that we cannot have multiple versions of the same component installed simultaneously. It also means that it is hard for multiple users to independently select, install and manage software.
>
> 类似的，RPM并不允许两个包含相同路径名称的软件包同时存在于系统中。
>
> ...
>
> 总体来说，这意味着我们不能同时安装相同组件的多个版本。这也意味着对于多用户系统来说，独立地选择、安装、管理软件并不容易。
>
> 引用自：The Purely Functional Software Deployment Model - Eelco Dolstra

并且，Windows和macOS也好不到哪里去。打开某些软件的安装目录，一大堆DLL库映入眼帘，令人感到恐惧。与Linux不同，Windows不倾向于使用包管理器，尽管这可能在一些程度上缓解了依赖问题，却还是禁不住磁盘占用率由于依赖库重复打包而飙升。

> Windows and Mac OS X tend to use monolithic deployment for applications: except for some large-grained dependencies on the operating system environment (e.g., on the kernel, the core GUI libraries, or Direct X), dependencies tend to be distributed as part of the application itself, with no sharing of dependencies between applications. 
>
> Windows与Mac OS X倾向于使用单体式部署————某些大型依赖除外（例：内核中的核心GUI库或Direct X），依赖库更倾向于被作为应用本身的一部分而分发，在应用之间不存在依赖共享。
>
> 引用自：The Purely Functional Software Deployment Model - Eelco Dolstra

而且，不仅仅是依赖问题，在开发软件时，很多开发者也肯定遇到过一套代码不能在两台机器上跑出同样效果的问题（比如：相对路径与绝对路径的使用）。

> The fact that a component works in a test environment, does not guarantee that it will work on a client site, since the test environment may provide components that are not explicitly declared as dependencies. As a consequence, component compositions are not **reproducible**. The installation of components in a common directory makes it hard to install multiple versions of a component or to roll back to a previous configuration if it turns out that upgrading produces a faulty configuration.
>
> 由于测试环境可能提供和被声明作为依赖的组件不同的依赖，一个组件能在测试环境中运行的事实，并不能保证它也能在客户端侧运行。结果就导致组件的组成**不可复现**。在公共目录中安装组件，如果升级导致错误的配置，安装多个版本组件以及回滚到先前的配置就显得尤为困难。
>
> 引用自：The Purely Functional Software Deployment Model - Eelco Dolstra

2006年，Eelco的博士论文《纯函数式包管理模型（The Purely Functional Software Deployment Model）》发布，这篇论文被视为是原子系统的“开山鼻祖”，在文中，他提出了一种极其前沿的包管理方案————Nix。与传统包管理器相比，Nix能够完美处理依赖与干扰问题，能够稳定复现，而且不会像其他包管理器那样“更新一半关机，再打开软件炸掉”的情形。

而这神奇的法术，又由什么驱动呢？

## 2.何为原子？

学过初中化学的各位都知道，原子（Atom）是化学反应中基本的单位。它难以拆分，难以被更改，没有“半成品”，只有存在或不存在这两种状态。强调这些概念可能听起来很奇怪，但这正是原子系统的核心————不可分割。用过Linux的同学都知道：在开机时，如果安装了多种Linux内核，我们可以在引导系统时自由切换，在安装、切换内核时，也基本不会影响其他软件包。我们对照原子性的定义来看看：编译后的内核不可拆分，切换内核也不可能只切一半————是的，这就是一个典型的原子操作。在原子系统中，我们可以将内核、操作系统、应用、数据比作相互包裹，密不可分的————洋葱？每次在更新系统或软件时，每一层的“核”都仅仅是被重新生成和更换。并且，在原子系统中绝对不会允许未完成的操作发生，不会出现“一颗老鼠屎坏了满锅粥”的情况。相比之下，给普通系统安装软件就像在一张由着上百张布料拼成的毯子上糊补丁————每次更换某块布料都可能对其上层或下层叠加的组件造成“毁灭性”打击，十分不稳定。

而前面所提到的某些厂商宣传的“原子”，本质上就是管理更精细些，然后取了个听起来高大上的名字，跟真正的原子操作可以说是相形见绌，八竿子打不着。

在这里其实也有一个误区：很多人会以为原子系统是“完全无法更改”的。这种说法并不严谨。比如NixOS会提供`/etc/nixos/configuration.nix`，能够方便快捷地调节系统大大小小的配置。甚至在你**完全掌握其语法参数**时，比通过其他途径更优雅。比如我自己就是一个不太“整洁”的人，每次装完一些软件包都会忘了卸载，而在NixOS里，这些“孤儿”就静静躺在配置文件里，可以随时删除，不会忘记，十分舒适。

不过，这种魔法，究竟是怎么实现的？

## 3.原子函数 (Nix)

> 注意！以下内容可能会略显枯燥，各位也可以直接跳转到视频最后一部分

> 由于本人未在其他发行版上使用过Nix，下文可能会出现Nix与NixOS混为一谈的现象，还请批评指正

NixOS以及Nix的工作原理主要在Eelco的这两篇论文中被完整阐释：
 - The Purely Functional Software Deployment Model
 - NixOS: A Purely Functional Linux Distribution
由于时间原因，我没有办法详细阅读两篇文章的全部内容，因而只阅读了其Introduction部分，具体的实现，感兴趣的各位也可以自行阅读余下的Foundations与Applications部分。在此我会尽可能简单地进行阐释，不过也可能出现疏漏，还请指正。

与其他发行版的分散式存储不同，在NixOS中，所有的软件包，无论是可执行文件还是源码，均存在于系统的“中央仓库”————`/nix/store`，由Nix统一进行管理，在这个目录下，每个软件都被当作一个“原子”。Nix并不关心这是什么原子，这个原子里存在什么，它只需要知道这个原子应该与哪些原子相关联。乍一看，这比普通的包管理器只是多了一个统一化管理。不过，让我们回看前面所提到的：它能够完美避免依赖冲突，这就又要提到Nix的另一精妙机制。在每个原子构建时都会被打上一个哈希值，这个哈希值由多种方式生成，和身份证号一样具有唯一性，不会出现撞车的问题，因而多种版本、多种来源的软件包就能够和谐共存。而且，这个机制能够强制要求所有原子在构建时指定依赖的完整路径，不会有找不到依赖的问题。而哈希串的唯一性也赋予了其快速精确查找的能力。Nix不需要去理解一个文件，只需要记录并扫描其依赖的哈希串就可以自动找到其依赖。和计算机编程中的函数思想完全一致，我不关心你是什么，我只关心我需要给你什么，你会给我吐出什么。简单，粗暴，优雅，快捷。

而至于NixOS，则比Nix更加激进。Nix仅仅是将软件层进行原子化，而NixOS将系统本身也作为原子，和软件包一样进行管理。NixOS继承了Nix“可多版本存在”的特性，而其版本在此被称为“世代”。不同的是，对用户来说，自己从“使用者”转变为了“构建者”，每次更改系统配置时都需要学习语法结构，自行修改配置文件，重新构建系统，学习难度直线上升。

不过，NixOS并不是原子系统唯一的出路。
