# 我理想中的操作系统，是什么样的？

> 声明：这绝不是一个Linux传教视频抑或是发行版推荐。

在当下操作系统日益多元化的时代，对于一个普通用户来说，要选择一个操作系统很简单，但要选择一个真正好用的操作系统却难于上青天。为此，我们不妨设想，如果世界上真的存在一个面向桌面用户的“完美”系统，它到底应该是什么样子的？

### 0.1 我们想要什么？

一个成熟，现代完善的操作系统，至少应该具备以下的特点：

- 稳定，无需为系统明天会不会坏掉而发愁。
- 安全，没有/很少会出现漏洞或恶意软件。
- 易用，开箱即用，容易上手
- 自由，任何人都可以添砖加瓦，无需为商业行为付费。
- 兼容，无需担忧自己的软件能否因为过时/适配而运行。

所以，Windows OUT（不稳定，不安全，不自由），macOS OUT（不易用，不自由，不兼容），剩下的只有Linux抑或是BSD。不过BSD的用户基数小，桌面适配远不及Linux，因而依然OUT。各位先不要着急喷诸如Linux上手难，全是黑框框之类的常见评论，尽管目前Linux桌面的体验有些糟糕，但这的的确确是在当下最合适的**内核**选择：Linux的自由度高，能够允许我们在其基础上作出较大的更改与创新，运行较为稳定，并且在开源的加持下，其安全性也基本能够有保障。

不过，众所周知，当前的Linux桌面发行版最大的问题就是“难用”，我之前出过视频讲这件事，并且连Linus自己都喷开发者“没有站在普通用户的视角”。所以，我在这里提出一些可能有利于改善这种问题的小设计，希望各位能够批评指正。

## 1. 原子化架构系统

### 1.1 系统层面

现代Linux的发行版拥有自由度高的特性，但是，对于普通用户来说，如此之高的自由度反而会给他们带来系统崩溃的风险，比如说杀手命令，误删文件无法找回等。所以，我们需要在这里引出一个概念——不可变系统 (Immutable Operating System)。不可变系统的核心就是系统文件只读，仅允许配置、数据文件可变。如果你看过我的上期视频，会发现：不可变这个概念，本身就是原子化在系统层面的一个表现（感谢@[Harold7_Zhang](https://space.bilibili.com/3110327)启发）。我们在网络上能够看到很多的电脑小白因为删除了系统文件而系统崩溃，但很少能够见到手机用户有类似的问题，本质就是“不可变”这个特性发挥了作用。并且这个特性根除了很多长此以往困扰Linux用户的问题，比如权限问题等。

所以，我在此总结：原子化就是Linux走向大众的最理想的选择。

那么说完了困境，让我们来看一些具体的方案。

- NixOS：将系统本身原子化，能够多世代共存，缺点是需要修改配置文件来修改配置，每次更新都需要重构系统。
- OSTree：以Git的思想管理系统，每次更新都进行增量远程拉取，能够多版本共存。
- A/B System Update：在一块硬盘上存储两个相同的系统核心，系统文件只读，每次都对非工作分区的系统应用更改，重启时切换分区以完成更新，缺点是会略微增加磁盘占用。

如果你做过安卓开发，你会很容易发现上面所提到的最后一项在本质上就是安卓的无缝更新或者OTA (Over-The-Air)。然而站在可靠性与易用性角度上，我们这里选择的应该是OSTree——在Linux上较为成熟，并且能够回滚。

### 1.2 软件层面

众所周知，Linux长期以来存在依赖地狱的困境（这是在我做的视频里第4次提到了），一个依赖的无法共存容易引发连锁效应，进而导致整个系统陷入“不能更新，不能安装，不能卸载”的局面，而应用层上的原子化恰恰能解决这个问题。

与前面相对应，软件层面也存在四种解决方案：

- 传统包管理器：依赖地狱的万恶之源，PASS
- Nix：将软件原子化，通过哈希值特性允许多版本软件包共存，从而规避依赖地狱，缺点是配置复杂。
- rpm-ostree（配套OSTree）：安装软件相当于在系统上“叠加”软件包，但无法完全规避依赖冲突，并且安装应用需要重启。
- 容器化（Flatpak，AppImage等）：通过不共享依赖的沙箱设计解决依赖问题。

稍微懂一点技术的老哥这个时候就会说了：这系统层面不应该是和软件层面的相互配套吗？怎么能错开用呢？没错，我们就是要通过这种错开的方式进行“取长补短”。根据我们要完全规避依赖冲突的目的，很明显我们应该选择Nix或者容器化。但我个人认为，此处应该以Nix为主，容器化为辅。

为什么？尽管Nix有配置复杂的缺点，但我们可以舍弃掉诸如Flakes花样用法，回归最原始的管理方式：只要nix-profile。这样一来，安装/卸载软件与普通包管理器的操作几乎无异，并且能够与应用商店互通，使用户操作更加便捷。不过Nix用户这个时候肯定又不乐意了：这不是摒弃了Nix的可重现性吗？是的，但是让我们仔细想想：对于一个普通用户来说，可重现性在日常的使用频繁吗？和纷繁复杂的语法比起来，是重新安装软件还是学习Flakes更难呢？对于企业用户来说，是写bash脚本一键安装简单，还是从头开始学习Nix Expression更简单？再者说，我们就不能像现在的各大手机厂商那样设计软件包迁移工具来帮助用户重装吗？

为什么不使用Flatpak？Flatpak的容器化设计固然很好，但是沙箱的特性使得它与宿主机互通的能力弱，比如QQ的剪贴板同步失效，拖拽文件不能识别等问题。而AppImage的管理方式不够优雅，并且会重复打包依赖，使得其出现不必要的空间浪费。

## 2. 优秀的容灾机制

我们都知道，在使用Linux时不免会遇到需要系统滚挂修复、调整分区等问题。这时候，大多数Linux用户的选择是：找到另一台设备->刷写LiveCD->回到原来的设备上进行操作，整个过程费时费力，那么我们能否简化这样的容灾机制？

熟悉Windows的各位都知道，在Windows设备的磁盘上有一块独特的分区——WINRE_DRV，这个分区存储的是Windows恢复环境，以备不时之需。然而恢复环境的系统是残血的，大部分管理功能被削减，只剩下必要的恢复功能。

受Windows启发，我理想的操作系统中也会有这样一个分区。不过相比与Windows，Linux的轻量特性赋予了其在有限空间内保留完整系统组件的能力，能够存储大量的系统管理应用。

但这并不完美。让我们假设一个极端的情况：假设，我们在调整系统时手滑将EFI格了怎么办？尽管有恢复环境存在，但其EFI文件依然和主系统存在同一个分区里，这个时候恢复环境反而成了累赘。所以请允许我作出一个反常规的操作：为恢复环境独立分出一个EFI分区。这样一来，即便是主EFI分区挂掉，我们依然能够进入恢复EFI分区修复系统。

示意图：

| 主EFI分区  | /boot              | /              | /var               | 恢复EFI分区  | RE（恢复环境） |
| :--------- | :----------------- | :------------- | :----------------- | :----------- | :------------- |
| 引导主系统 | 主系统启动所需文件 | 根分区（只读） | 配置、用户数据分区 | 引导恢复环境 | 用于修复系统   |

那么问题来了：现代BIOS是否允许这样的操作？答案是允许，但不推荐。UEFI规范中并没有强制规定只允许一个EFI分区，但是这样做可能会导致**引导混乱**。并且，Windows明确表示**不支持多ESP分区**，但Linux允许。因而，这种机制可能只能存在设想阶段了。但话又说回来，毕竟是**理想中**的操作系统嘛，有点不现实的成分也正常，所以这个设计更像是一个可选项。

## 3. LSW (Linux Subsystem Windows)

2021年6月，Microslop宣布推出WSA (Windows Subsystem Android)，和先前已经推出的WSL (Windows Subsystem Linux) 一样用于适配安卓应用。尽管这个功能很不错，但实际上普通用户所用到的机会并不多，Windows本身的软件生态足够完善，没有必要使用Android/Linux来弥补。

不过在Linux上可就不一样了。对于Windows用户来说，迁移到Windows最大的难点就是软件适配。尽管很多软件已经响应号召，有了官方包或者Flatpak，但很多专业软件并没有。尽管目前存在Wine和虚拟机等解决方案，但这并不够。Wine对于Windows API的复现性不足，部分软件依然无法正常运行，传统虚拟机没有做到无缝集成，对用户来说不够优雅。
所以我们在此引入LSW (Linux Subsystem Windows) 这一概念。与Windows一样，我们采用虚拟化技术来创建一个”子系统“。这样一来，我们既摆脱了兼容性问题，也能够以优雅的方式运行Windows软件。
自然而然，对于这个解决方案下的项目，我们会想到WinBoat，它以Docker为容器，通过RDP（远程桌面连接）连接本地虚拟机来实现无缝集成。不过很可惜，这个项目目前还是Beta阶段，并没有加入诸如GPU支持与透传这类功能，只能在此期望越来越好吧。

## 4. 更友好的用户体验

现代Linux大多采用Grub作为引导器，Grub很好，但每次通过Grub启动时都需要先看几秒钟的黑框框，十分不友好。所以我个人更倾向于Grub静默引导，而不是要提供一个菜单可供选择。这个时候有的同学就要问了：那我怎么进入rescue环境？我怎么切换内核？怎么回滚？还记得我们前面提到的双引导分区设计吗？我们完全可以把这些普通用户日常不需要的功能放在恢复EFI分区中作为高级选项使用。并且，在系统启动失败时，系统不应该直接进入initramfs这样的终端环境，而是首先询问用户是否希望重启引导到恢复环境中，通过chroot进行系统修复。

镜像源自动测速是我认为Arch Linux为数不多的好设计，它通过后台服务的形式对系统预置的镜像源列表进行测速，从而选出最佳的路线，这对于大陆用户来说十分有用，但不知为何其他发行版似乎并没有类似的设计。
